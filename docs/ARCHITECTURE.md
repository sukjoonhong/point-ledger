# Point Ledger System: Architecture & Design Decisions

본 문서는 대규모 트래픽 처리와 데이터 결함 제로(Zero-defect)를 목표로 하는 Point Ledger System의 핵심 설계 원칙 및 기술적 의사결정 사항을 기록합니다.

---

## 1. 아키텍처 설계 원칙 (Architecture Principles)

### 1.1 원장 중심 설계 (Ledger-first Design)
포인트 시스템의 가장 신뢰할 수 있는 데이터의 원천(Source of Truth)을 거래 원장에 둡니다. 잔액(Balance)은 원장으로부터 계산 및 동기화되는 파생 상태(Derived State)로 취급하며, 모든 상태 변화는 원장의 기록을 통해 증명되어야 합니다.

### 1.2 물리적 외래키(FK) 제약조건 회피
본 시스템은 연간 수조 원 규모의 트래픽과 DB 샤딩(Sharding) 환경을 전제로 설계되었습니다.
- 데이터베이스 부하 최적화: 물리적 외래키는 분산 데이터베이스 환경에서 데이터 삽입 시 참조 무결성 확인을 위한 추가적인 I/O와 락(Lock) 경합을 유발합니다. 이는 시스템의 수평적 확장을 가로막는 주요 병목이 됩니다.
- 논리적 무결성: 시스템은 물리적 제약조건 대신 인덱스 최적화와 애플리케이션 레이어(PointLedgerService)에서의 철저한 비즈니스 유효성 검증을 통해 무결성을 확보합니다.

### 1.3 단일 아티팩트 기반의 멀티 역할 배포 (Multi-Role Deployment)
동일한 코드베이스를 유지하되, 실행 시 주입되는 스프링 프로파일에 따라 서버의 역할을 유연하게 결정합니다.
- 역할 분리 (API, Worker, Scheduler): 각 노드는 배포 시점에 지정된 역할(Profile)에 필요한 Bean들만 로드합니다. 이는 리소스 점유를 최적화하고 불필요한 엔드포인트 노출을 차단합니다.
- 장애 전파 차단: 워커 노드의 CPU/Memory 고갈이 사용자 API 노드의 가용성에 영향을 주지 않도록 물리적 실행 환경을 격리합니다.
- 운영 효율성: 단일 아티팩트로 관리되므로 노드 간 비즈니스 로직 버전 불일치 리스크가 없으며, 특정 역할의 노드만 개별적으로 스케일 아웃(Scale-out)이 가능합니다.

## 2. 클라우드 인프라 구성도 (AWS Diagram)

본 프로젝트는 연간 5조 원 규모의 트래픽을 처리하는 상용 서비스 환경을 전제로 설계되었으며, 이를 지원하는 AWS 인프라 아키텍처는 다음과 같습니다. H2 파일 기반의 로컬 환경과 달리, 프로덕션 환경에서는 Amazon RDS를 사용하여 원장의 신뢰성을 극대화합니다.

![AWS Infrastructure Diagram](src/main/resources/static/aws.png)

### 2.1 인프라 핵심 설계 포인트

- **컴포넌트별 역할 및 연결**:
   - **API 레이어 (EC2/ECS, `api` Profile)**: 사용자 요청을 접수하고 멱등성 검증 후 트랜잭션을 원장(RDS)에 기록합니다. 이후 작업을 `Amazon SQS`에 인큐잉(Enqueue)하여 워커로 위임합니다.
   - **워커 레이어 (EC2/ECS, `worker` Profile)**: `Amazon SQS`로부터 메시지를 수신하여 실제 지갑(Wallet) 잔액을 변경하는 고부하 로직을 수행하고 그 결과를 원장(RDS)에 최종 반영합니다.
   - **Amazon SQS**: API 서버와 워커 서버를 디커플링(Decoupling)하여 대규모 트래픽 폭주 시에도 API의 가용성을 보장하고, 워커 서버는 자신의 처리 속도에 맞춰 작업을 안정적으로 수행합니다. (Dead Letter Queue를 통해 처리 실패에 대한 재시도 및 안정성을 강화합니다.)
   - **Amazon RDS (Aurora PostgreSQL)**: 시스템의 유일한 **'Source of Truth(신뢰할 수 있는 단일 원천)'**인 거래 원장(Ledger) 데이터베이스입니다. 로컬 환경의 H2 파일을 대체하여 영속성과 트랜잭션 원자성을 보장합니다.

이와 같이 **`spring.profiles.active`를 통해 `api`와 `worker` 역할을 물리적으로 분리된 EC2 인스턴스나 ECS 컨테이너에 배포**함으로써, 우리가 의도한 물리적 격리와 개별 스케일 아웃(Scale-out)이 AWS 환경에서 완벽하게 구현됩니다.

## 3. 엔터프라이즈급 기술적 의사결정 (ADR)

### 3.1 Balance-Neutral RE_EARN 로직
포인트 사용 취소 시, 이미 만료된 자산에 대한 보상 처리(RE_EARN)는 지갑 잔액에 영향을 주지 않는 '중립적' 방식으로 설계되었습니다.
- 설계 의도: CANCEL_USE(사용 취소) 단계에서 이미 전체 취소 금액이 지갑 잔액에 선행 합산됩니다.
- 선형성 보장: RE_EARN 단계에서는 잔액을 중복 가산하지 않고, 오직 '만료 자산을 대체하는 신규 자산 생성'만을 원장에 기록합니다. 이는 지갑 잔액의 정합성을 유지하면서도 원장 시퀀스의 선형성(Linearity)과 추적성을 완벽하게 보장하기 위함입니다.

### 3.2 3중 멱등성 방어 (3-Layer Idempotency)
중복 요청으로 인한 자산 오적립을 방지하기 위해 3단계 방어막을 구축했습니다.
- L1 (Local Lock): Caffeine Cache를 활용하여 Hot-key 중복 요청을 애플리케이션 입구에서 1차 차단합니다.
- L2 (Distributed Lock): 분산 락 추상화를 통해 여러 서버 노드 간의 동일 자산 접근을 제어합니다.
- L3 (Database Unique Index): 최후의 보루로 DB 유니크 제약 조건을 사용하여 물리적 중복 삽입을 원천 차단합니다.

### 3.3 Transactional Outbox Pattern
비즈니스 로직 성공과 이벤트 발행의 원자성(Atomicity)을 보장합니다. 비즈니스 DB 트랜잭션 내에 이벤트를 point_outbox 테이블에 함께 저장하고, 별도의 릴레이 서비스가 이를 확인하여 발행함으로써 메시지 유실 없는 At-Least-Once Delivery를 보장합니다.

### 3.4 시퀀스 번호 및 자가 치유(Self-healing) 리플레이
분산 환경에서 이벤트 순서 뒤바뀜이나 유실 시나리오에 대응합니다.
- 선형성 유지: 지갑별 시퀀스 번호를 엄격히 체크하여 유실(Gap) 발생 시 처리를 중단하고 복구 모드로 전환합니다.
- 리플레이: 누락된 시퀀스가 감지되면 원장 데이터를 다시 조회하여 상태를 재현함으로써 지갑과 원장 간의 정합성을 강제로 일치시킵니다.

## 4. API Specification: 포인트 이벤트 발행

- URL: POST /v1/points/enqueue
- Status: 202 Accepted
- 비고: 상위 도메인으로부터 확정된 거래 원장 데이터를 수집하며, 비동기 파이프라인을 통해 처리됩니다.

[Request Body Example]
```json lines
{
   "memberId": 1001,
   "amount": 5000,
   "pointKey": "REQ_EARN_001",
   "originalPointKey": null,
   "type": "EARN",
   "sequenceNum": 1
}
```

## 5. 도메인 규칙 및 정책

1. 사용 우선순위
   - 관리자 수기 지급(ADMIN) 포인트 최우선 차감.
   - 유효기간이 짧게 남은 자산(Early Expire) 우선 차감 (FIFO/LIFO 복합 적용).
2. 환불 정책
   - 활성 상태의 자산은 기존 자산의 잔액으로 복구(Restore).
   - 이미 만료된 자산은 신규 보상 적립(RE_EARN)으로 대체 발행하여 원장 기록 분리.
3. 제약 조건
   - 적립 한도: 정책 설정에 따른 1회 최대 적립액 및 지갑 총 보유 한도 제한.
   - 취소 제한: 이미 소모된 적립 자산은 원본 적립 취소가 불가능하며, 사용 취소 로직을 선행해야 함.
